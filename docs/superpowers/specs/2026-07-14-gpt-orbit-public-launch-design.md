# Gpt Orbit 公开发布与社交传播设计

## 目标

把已经完成并在 Windows 11 上验收的 Gpt Orbit 周额度悬浮组件整理为可公开访问、可下载、可继续开发的 GitHub 项目，并为小红书和抖音制作统一但适配各平台的首发素材。

发布成功的判断标准：

- GitHub 上存在公开仓库，源码、中文 README、许可证和版本说明可以正常访问。
- GitHub Release 提供可下载的 Windows x64 安装包，并显示文件校验值。
- 仓库清楚说明工具用途、安装方式、系统要求、隐私边界和已知限制。
- 小红书和抖音各有一张符合平台阅读习惯的竖版封面。
- 两个平台各有一套可直接复制发布的中文文案和明确的发布步骤。

## 产品定位

Gpt Orbit 是一个 Windows 11 桌面悬浮小组件，用于实时查看 Codex 周额度。核心展示包括周剩余百分比、周重置倒计时和剩余重置次数；同时支持桌面置顶、自由拖动、悬停展开、托盘运行和位置记忆。

首发传播不介绍尚未稳定的扩展设想，也不把产品包装成官方 OpenAI 或 Codex 应用。对外统一表述为个人制作的免费开源桌面工具，并明确它与 OpenAI 无官方关联。

## GitHub 发布设计

### 仓库与许可

- 建议仓库名：`gpt-orbit`。
- 仓库可见性：公开。
- 源码和安装包均公开。
- 许可证：MIT，方便学习、修改和二次开发，同时保留作者与许可证声明。
- 默认分支：`main`。

### 仓库首页

README 以中文为主，按以下顺序组织：

1. 产品名称、一句话定位和最终效果图。
2. Windows 安装包下载入口。
3. 核心功能与交互说明。
4. 系统要求、安装和卸载方法。
5. 数据来源、隐私说明与非官方声明。
6. 本地开发、测试和构建方法。
7. 已知限制与后续计划。
8. 许可证。

README 不暴露本机绝对路径、账户信息、登录 URL、Token、调试日志或其他私密数据。

### 版本发布

- 首个公开版本使用 `v0.1.0`。
- GitHub Release 标题使用 `Gpt Orbit v0.1.0`。
- Release 附带 Windows x64 NSIS 安装包和 SHA-256。
- 发布说明写清主要功能、安装方式、系统要求、已知限制和非官方声明。
- 安装包不提交到 Git 历史，只作为 Release 资源上传。

### 自动构建

使用 Tauri 官方 `tauri-action` 工作流。在版本标签发布时，Windows Runner 安装 Node 与 Rust 依赖，执行前端测试、Rust 测试和 Weekly 版本构建，再把安装包上传到 GitHub Release。工作流权限只开放发布所需的 `contents: write`。

第一版允许先上传已经在真实 Windows 环境验收的本地安装包，再让自动构建工作流作为后续版本的可复现发布通道；两者的版本、文件名和校验值必须一致或在发布说明中明确区分。

## 视觉传播设计

采用“产品展示型”作为主方向，辅以个人开发故事。视觉中心始终是真实 Gpt Orbit 组件，不使用与实际产品不一致的概念界面。

### 统一视觉语言

- 背景：克制的深色 Windows 桌面氛围，保留桌面工具的真实使用语境。
- 主体：放大的周额度悬浮球，周围带轻微紫蓝玻璃光晕。
- 色彩：深蓝、靛紫、冰蓝，红紫色重置次数徽标作为视觉锚点。
- 字体：高可读的现代中文无衬线字体，避免细字、拥挤字距和大段说明。
- 风格：精致、轻量、真实可用；避免夸张霓虹、游戏 HUD 和与成品不符的视觉效果。
- 品牌边界：不使用 OpenAI 官方徽标，不暗示官方出品。

### 小红书封面

- 比例：3:4 竖版。
- 主标题：`我做了个 Codex 额度悬浮球`
- 副标题：`周额度终于能一眼看懂了`
- 构图：标题位于上部，放大的真实组件位于中部偏下，底部以小标签呈现 `Windows · 免费开源`。
- 目标：用户在信息流中两秒内明白“这是一个显示 Codex 周额度的 Windows 小工具”。

### 抖音封面

- 比例：9:16 竖版。
- 主标题：`Codex 周额度悬浮球`
- 辅助信息：`Windows 免费开源`
- 构图：文字更少、更大；组件位于画面视觉中心，预留上下安全区，避免被头像、标题和操作按钮遮挡。
- 目标：在手机小尺寸预览中仍能认出产品和核心用途。

## 文案设计

### 小红书

采用“痛点—解决办法—功能—开源下载—互动”的结构。语气真实、有个人开发感，不夸大效果。正文包含实际功能、适用系统、GitHub 下载位置、非官方说明和邀请反馈。话题标签控制在与 Windows、Codex、桌面工具、开源项目直接相关的范围内。

### 抖音

采用更短的口语化结构。开头一句直接说明“为了不用反复找 Codex 周额度，我做了这个悬浮球”，随后用三至四个短句说明周额度、倒计时、重置次数和免费开源。结尾引导用户通过主页或置顶评论访问 GitHub，具体方式以账号可用的链接能力为准。

## 发布顺序

1. 清理公开仓库内容，补齐 README、LICENSE、截图和自动构建工作流。
2. 在本地完整运行测试和生产构建，核对安装包与校验值。
3. 创建公开 GitHub 仓库，推送默认分支。
4. 创建 `v0.1.0` Release 并上传安装包。
5. 打开公开页面和 Release 下载链接做最终验证。
6. 生成并检查小红书与抖音封面，保证文字准确、主体清晰、安全区合理。
7. 先发布小红书图文，再发布抖音图文或短视频；发布后检查封面裁切、链接可达性和评论区引导。

## 安全与质量边界

- 发布前扫描源码、文档和 Git 历史中的凭据、账户信息、绝对路径和个人数据。
- 不把本地缓存、构建目录、测试输出、临时截图和调试日志提交到公开仓库。
- 不声称已经验证未执行的系统、DPI、卸载或多显示器场景。
- 安装包与源码版本必须能对应；Release 必须提供校验值。
- 封面文字必须逐字检查，不能依赖图片模型准确生成关键中文；必要时使用确定性的本地排版覆盖文字。

## 验收清单

- 公开仓库和 Release 在未登录状态下可以访问。
- README 的下载入口能到达正确版本。
- Windows 安装包可下载，文件大小和 SHA-256 与发布记录一致。
- 自动构建工作流语法有效，所用 Action 固定在明确版本。
- 仓库中没有本机路径、凭据、Token 或不应公开的输出。
- 两张封面分别为 3:4 与 9:16，中文无错字，产品主体清晰。
- 小红书和抖音文案可直接复制，不包含未替换占位符。
- 发布指南明确说明封面选择、正文粘贴、链接放置和发布后检查步骤。
